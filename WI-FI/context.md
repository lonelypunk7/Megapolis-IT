# Wi-Fi Captive Portal — контекст проекта

> Здесь будет собираться информация по Captive Portal проекту.

## Роль QA
Пользователь тестирует Captive Portal — веб-портал авторизации для Wi-Fi доступа.

## Архитектура и функционал

### Список профилей
- Список профилей (каптив-порталов) с фильтрацией:
  - Поиск по **наименованию**
  - Фильтр по **оператору**
  - Фильтр по **дате**
  - Фильтр по **статусу**
- В списке отображается:
  - Наименование
  - Статус
  - Кнопка перехода в карточку

### Создание нового профиля
- **Наименование** * (обязательное)
- **Даты действия** (период активности)
- **Версия** * (обязательная)
- **Выбор языка**: RU, EN * (обязательный)

#### Визуальная настройка (необязательные)
- Загрузка **логотипа**
- Выбор **цветового решения**

#### Статус (автоматически по датам действия)
- Черновик
- Действует
- План
- Завершён

#### Предпросмотр
- Рядом с формой — live preview того, что увидит пользователь

#### Пользовательское соглашение
- Текст: «Я принимаю условия [Пользовательского соглашения]»
- Кнопка перехода в пользовательское соглашение

#### Загрузка оферты
- Возможность загрузить файл расширения: **PDF, HTML, DOCX**

### Белый список ресурсов
- Добавление ресурсов каждого вида (до 20 шт.)
- Поля для каждого ресурса:
  - **Тип**: домен, IP-адрес, маска домена, подсеть
  - **Значение**
  - **Описание** (например: «ВКонтакте»)

#### Форматы значений и саджесты
| Тип | Формат | Пример |
|-----|--------|--------|
| Домен | `example.ru` | `example.ru` |
| Маска домена | `*.example.ru` | `*.example.ru` (звёздочка = любой поддомен) |
| IP-адрес | `192.168.1.1` | `192.168.1.1` |
| Подсеть | CIDR | `192.168.0.0/16` |

- Итог — таблица сгруппированная по типам с указанием значения и описания

### Редактирование профиля
- Редактирование всех полей карточки
- **Удаление** — с модалкой подтверждения
- Кнопки **ручной активации** и **завершения действия** профиля (меняют статус)

### Привязка логической сети
- Отдельная форма для привязки логической сети
- После привязки — **поле Оператора** подтягивается автоматически к профилю
- Оператор определяется через привязку логической сети → объектам размещения

## Flow авторизации (User Journey)

### Основной позитивный сценарий
1. Пользователь подключается к гостевой Wi-Fi сети
2. Точка доступа перенаправляет браузер на страницу портала
3. Captive Portal запрашивает настройки и доступные способы авторизации
4. Backend ЕОС передаёт в HTML список доступных способов авторизации + параметры отображения страницы (настройки оператора, логическая сеть, объект размещения, точки доступа)
5. Пользователь нажимает **«Войти по MosId»**
6. Frontend перенаправляет на страницу авторизации mos.ru
7. Mos.ru проверяет авторизацию (запрашивает логин/пароль)
8. Backend запрашивает данные пользователя
9. Пользователь вводит верные логин и пароль → успешно авторизуется в MosID
10. Backend сопоставляет MAC-адрес от mos.ru с MAC-адресом, хранящимся в кэше ЕОС
11. При успешной проверке (**200**) — открывается доступ к трафику
12. Frontend перенаправляет на страницу авторизации Captive Portal → пользователь может запрашивать интернет-ресурсы

## Flow авторизации — детальные сценарии

### 1. Подключение к Wi-Fi и открытие портала
- Пользователь подключается к гостевой Wi-Fi сети
- Точка доступа перенаправляет браузер на страницу портала
- Captive Portal запрашивает настройки и доступные способы авторизации
- Backend EОС: `GET /radius/authorize` — передаёт в HTML список доступных способов и параметры отображения страницы (из настроек оператора, логической сети, объекта размещения, точек доступа)

### 2. Пользователь нажимает «Войти по MosId»
- Backend: `POST /sudir/authorize`
- Frontend перенаправляет пользователя на страницу авторизации mos.ru

### 3. Mos.ru возвращает 200 — проверяет авторизацию
- Mos.ru запрашивает логин и пароль
- Пользователь вводит верные логин и пароль, успешно авторизуется в MosID
- Backend отправляет запрос `GET /sudir/callback` и сопоставляет MAC-адрес от mos.ru с MAC-адресом, хранящимся в кэше
- В случае успешной проверки (код **200**) — открывается доступ к трафику
- Frontend перенаправляет на страницу авторизации Captive Portal — пользователь может запрашивать интернет-ресурсы

### 4. Mos.ru возвращает 400 — ошибка
- Backend EОС: **400** (например, не парсится MAC-адрес или пустой параметр)
- Frontend Captive Portal: выводит ошибку «Авторизация не пройдена»

### 5. Mos.ru возвращает 401 — неверный пароль
- Backend EОС: **401**
- Frontend Captive Portal: отображает ошибку «Авторизация не пройдена»

### 6. Mos.ru возвращает 403 — доступ запрещён
- Причина: VPN включен, неактивна точка доступа, нет контракта и пр.
- Backend EОС: **403** — «Доступ запрещён»
- Frontend: выводит ошибку «Доступ запрещён»

### 7. Пользователь закрыл окно авторизации mos.ru
- Backend: **499** — «Авторизация не пройдена»
- Frontend: отображает ошибку «Авторизация не пройдена», выводит кнопку перехода к способам авторизации

### 8. Mos.ru «лежит» — HTTP 500
- Frontend: выводит ошибку «Сервис недоступен»
- Пользователь имеет возможность выбрать другой способ авторизации

## Негативные сценарии

| Сценарий | Ответ источника | Код Backend ЕОС | Сообщение пользователю |
|----------|-----------------|-----------------|------------------------|
| Неверный логин/пароль | Mos.ru → **400** | **401** | «Авторизация не пройдена» |
| Доступ запрещён (нет контракта, неактивна точка и пр.) | Mos.ru → **403** | **403** | «Доступ запрещён» (с бэка) |
| Пользователь закрыл окно авторизации | Backend → **499** | — | «Авторизация не пройдена» + кнопка возврата к способам авторизации |
| Mos.ru «лежит» | Mos.ru → **500** | — | «Сервис недоступен» + возможность выбрать другой способ авторизации |

## Глоссарий

### Основные понятия

| Термин | Определение |
|--------|-------------|
| **Авторизация** | Процесс принятия решения о предоставлении пользователю доступа в интернет или к определённым ресурсам |
| **Аутентификация** | Процесс подтверждения пользователя (например, через SMS или посредством OpenID) |
| **Идентификация** | Процесс определения пользователя, например, по номеру телефона/устройству, при попытке подключения |
| **Пользователь** | Физ. лицо, подключающееся к городской Wi-Fi сети для получения доступа к интернету |
| **Устройство пользователя** | Техническое средство: смартфон/планшет/ноутбук, с которого пользователь подключается к Wi-Fi сети |

### Сеть и оборудование

| Термин | Определение |
|--------|-------------|
| **ТД (Точка доступа)** | Сетевое устройство с конкретной физической локацией, обеспечивающее пользователям беспроводное подключение к Wi-Fi сети. Привязана к объекту размещения |
| **Объект размещения** | Физическая локация, на которой установлены одна или несколько точек доступа Wi-Fi. Имеет адрес, координаты |
| **Зона покрытия** | Территория, в пределах которой возможно подключение к конкретной точке доступа / группе точек |
| **Логическая сеть** | Виртуально выделенная сеть, объединяющая точки доступа и сервисы в рамках заданных правил маршрутизации, безопасности и управления доступом. Это идентификатор беспроводной сети, которую видит пользователь при выборе Wi-Fi на своём устройстве |
| **SSID** | Идентификатор логической сети (Service Set Identifier) |
| **WLC / AP** | Контроллер беспроводной сети (Wireless LAN Controller) / Точка доступа (Access Point) |
| **EoC (Ethernet over Coax)** | Технология передачи данных (интернета), использующая коаксиальные кабели кабельного телевидения |
| **Passpoint** | Стандарт для автоматического подключения к Wi-Fi (Hotspot 2.0), также используется как защищённый профиль |

### Сессии и доступ

| Термин | Определение |
|--------|-------------|
| **Сессия доступа** | Период времени, в течение которого пользователь подключён к Wi-Fi сети после успешной аутентификации |
| **Реестр сессий** | Интерфейс отображения активных и завершённых сессий с основными параметрами (время, трафик, причина завершения, применённые правила) |
| **Повторная аутентификация** | Повторная проверка пользователя при истечении сессии или изменении условий доступа |
| **Политика доступа** | Совокупность правил, условий, определяющая, какой доступ с какими параметрами предоставить пользователю или устройству при подключении к Wi-Fi |
| **Контекст авторизации** | Набор параметров, определяющих организацию-оператора, объект размещения, логическую сеть (SSID), точку доступа и характеристики устройства — используется для выбора корректных политик |

### MAC-адрес и устройства

| Термин | Определение |
|--------|-------------|
| **MAC-адрес** | Аппаратный адрес устройства (Media Access Control) |
| **Рандомизированный MAC-адрес** | Случайно генерируемый MAC-адрес, при котором на устройство передаётся не свой реальный постоянный адрес, а случайно сгенерированный. Может меняться раз в сутки. Создаёт проблемы при повторной аутентификации |
| **Случайно генерируемый MAC-адрес** | Технология приватности, при которой вместо реального MAC-адреса передаётся случайный |

### Организационные понятия

| Термин | Определение |
|--------|-------------|
| **Организация-оператор** | Юр. лицо, которому принадлежит право на предоставление услуг Wi-Fi на закреплённых за ней объектах размещения. Управляет своими точками доступа, настраивает параметры авторизации (способы входа), визуальное оформление портала (логотипы, цвета, тексты), получает статистику и отчёты по своим объектам, но не имеет доступа к данным других организаций |
| **Объект размещения** | Физическая локация, на которой установлены одна или несколько точек доступа Wi-Fi. Имеет адрес, координаты |
| **Заказ** | Номер заказа загрузки файла. Заказ на отключение или подключение ТД, корректирующий или акт БУУ, уникален |

### Аутентификация и способы входа

| Термин | Определение |
|--------|-------------|
| **Одноразовый код (SMS OTP)** | One-Time Password — временный числовой и буквенно-цифровой код, отправляемый пользователю через SMS-сообщение на указанный им номер телефона |
| **ЕСИА** | Единая система идентификации и аутентификации (Госуслуги). Единый логин и пароль («ключ»), позволяющий авторизоваться на различных порталах, не создавая новые учётные записи |
| **СУДИР** | Система управления доступом к информационным ресурсам города Москвы (MosID) |
| **ГИС ЕМП** | Государственная информационная система «Единый мобильный портал» (приоритетный SMS-шлюз для городской сети) |
| **Captcha** | Тест для защиты от автоматических действий (упоминается как мера при превышении лимитов попыток) |

### Способы авторизации

| Способ | Описание | Источник |
|--------|----------|----------|
| **MosID (СУДИР)** | Авторизация по логину/паролю через mos.ru | СУДИР (MosID) |
| **Госуслуги (ЕСИА)** | Авторизация через единый логин/пароль Госуслуг | ЕСИА |
| **Яндекс** | Авторизация через аккаунт Яндекс | Яндекс |
| **SMS OTP** | Одноразовый код по SMS на телефон | ГИС ЕМП / оператор |
| **ЕОС** | Внутренняя авторизация через базу ЕОС | ЕОС |

> ⚠️ Источник блокировки: ЕОС, СУДИР, Госуслуги, Яндекс — пока непонятно, можем ли получать данные о блокировке от способов авторизации

### Архитектура и инфраструктура

| Термин | Определение |
|--------|-------------|
| **AAA-платформа** | Система аутентификации (authentication), авторизации (authorization) и учёта (accounting) |
| **RADIUS** | Протокол удалённой аутентификации и учёта (Remote Authentication Dial-In User Service) |
| **LDAP** | Lightweight Directory Access Protocol — служба каталогов для централизованного хранения учётных записей |
| **UDP** | Протокол пользовательских датаграмм (используется для RADIUS-трафика) |
| **CAPTCHA** | Тест для защиты от автоматических действий |
| **ЦОД** | Центр обработки данных |
| **Репликация данных** | Автоматическое дублирование конфигурационных и учётных данных между несколькими ЦОД для обеспечения согласованности и восстановления после отказов |
| **Отказоустойчивость** | Способность системы сохранять работоспособность при отказе отдельных компонентов или целого ЦОД за счёт репликации данных, резервирования и автоматического переключения |
| **Балансировка нагрузки** | Распределение внешнего трафика (HTTP, RADIUS) между экземплярами сервисов для обеспечения отказоустойчивости и производительности |
| **Шаблон конфигурации** | Набор готовых настроек (политики, порталы, интеграции), который можно применить ко многим объектам системы (организациям, объектам размещения, точкам доступа) без ручного конфигурирования каждого |
| **Трассировка запросов** | Механизм генерации уникального идентификатора (Trace ID) для каждого запроса, фиксация всех шагов его обработки с временем выполнения — используется для анализа производительности и поиска ошибок |
| **SDK** | Набор средств разработки (Software Development Kit) |
| **SLA** | Соглашение об уровне обслуживания (Service Level Agreement) |
| **СМКСС** | Система мониторинга качества сопровождения систем |
| **OPM** | Обеспечение требований к оперативно-розыскным мероприятиям (из контекста «передача идентификационных данных для обеспечения ОРМ») |
| **IPFIX** | Протокол экспорта информации о сетевых потоках (IP Flow Information Export). Детализированные данные о сетевом трафике (кто, куда, какой протокол, объём), передаваемые от операторов связи и сохраняемые в аналитическом хранилище |

### Данные и аналитика

| Термин | Определение |
|--------|-------------|
| **Агрегация данных** | Процесс укрупнения детализированных показателей (например, посекундных в почасовые) для долгосрочного хранения и анализа тенденций |

| **НСИ** | Нормативно-справочная информация |
